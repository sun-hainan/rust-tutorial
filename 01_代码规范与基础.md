## 1. 注释类型与编译器行为

### 生活类比 🎯

> 注释就像书的批注——读者看得见，但不影响正文内容。编译器也一样：注释只是给人类看的"笔记"，机器完全忽略。

### 概念讲解

Rust 有三种注释：

| 类型 | 语法 | 编译器行为 |
|------|------|-----------|
| 行注释 | `//` | 完全忽略，编译后不生成任何目标码 |
| 块注释 | `/* */` | 完全忽略 |
| 文档注释 | `///` 或 `//!` | 编译进 crate 文档（rustdoc） |

从编译原理看：注释是词法分析阶段的"令牌"，最终不生成任何目标码。从机器层面看：注释在编译后完全消失，不影响二进制体积。

### 代码示例（逐行中文注释）

```rust
// ============================================================
// 模块一：代码规范与基础
// 注释 / 缩进 / 标识符 / 关键字
// ============================================================

/// 这是一个文档注释，可以描述下面的函数
/// 用于生成 API 文档（类似 Javadoc / Python docstring）
fn greet(name: &str) -> String {
    format!("Hello, {}!", name) // 行内注释放行尾
}
```

### 常见错误

```rust
// ❌ 错误：在字符串中写 // 不会产生注释
let s = "// 这不是注释"; // 这是字符串

// ✅ 正确：多行注释可以跨越行
let x = 1 /* + 2 */ + 3; // 结果为 4
```

## 2. 缩进规范与标识符规则

### 生活类比 🎯

> 标识符命名就像给人起名字——你要告诉编译器"这是谁"。Rust 有一套命名规范，就像中文的姓氏 + 名字，让代码更易读。

### 概念讲解

**缩进：** Rust 编译器不强制缩进大小，但社区规范使用 4 空格。缩进影响词法分析器对"缩进级别"的判定。

**标识符命名规则：**
- 由字母、数字、下划线组成
- 不能以数字开头
- 区分大小写

**Rust 关键字（reserved keywords）：** 不能用作标识符

### 代码示例（逐行中文注释）

```rust
// 合法标识符
let snake_case = 1;    // 小写蛇形（Rust 标准风格）
let camel_case = 2;     // 驼峰命名
let _private_var = 3;   // 下划线开头表示未使用警告抑制
let __dunder = 4;      // 双下划线保留给编译器/标准库

// 非法：不能以数字开头
// let 2fast = 5;      // ❌ 编译错误

// 关键字不能用作标识符
// let fn = 5;         // ❌ 编译错误

let r#return = 7;      // ✅ 转义关键字（raw identifier）
println!("r#return = {}", r#return);
```

## 3. 原始标识符（raw identifier）

### 生活类比 🎯

> raw identifier 就像给关键字起别名——当你必须和一个叫"fn"的人说话时，你可以叫他"r#fn"来避免混淆。

### 概念讲解

当与旧版 Rust 或其他语言交互时，某些版本的关键字可能与我们的标识符冲突。Rust 允许使用 `r#` 前缀将关键字转义为普通标识符。

**使用场景：**
- FFI 调用 C 函数时，C 函数名恰好是 Rust 关键字
- 老版本 crate 升级后新关键字与你的变量名冲突

### 代码示例

```rust
fn raw_identifier_demo() {
    // Rust 2015 中 "dyn" 不是关键字
    // Rust 2018+ 中 "dyn" 成为关键字
    let r#dyn = "dynamic"; // 绕过关键字冲突
    println!("raw dyn = {}", r#dyn);
}
```

## 4. 命名规范

### ASCII 图示

```
┌─────────────────────────────────────────────────────────┐
│  Rust 命名规范                                           │
├─────────────────────────────────────────────────────────┤
│  变量 / 函数 / 方法  →  snake_case（蛇形）              │
│  类型 / Trait / 结构体 → PascalCase（驼峰）            │
│  常量（const）     →  SCREAMING_SNAKE_CASE（全大写）    │
│  生命周期          →  'a, 'static（小写字母前缀）       │
└─────────────────────────────────────────────────────────┘
```

### 代码示例

```rust
fn naming_conventions_demo() {
    // 变量：snake_case
    let max_buffer_size = 1024;

    // 常量：SCREAMING_SNAKE_CASE
    const MAX_RETRY_COUNT: u32 = 3;

    // 类型：PascalCase
    struct UserProfile { name: String, age: u32 }

    // Trait：PascalCase
    trait Serializable { fn serialize(&self) -> Vec<u8>; }

    // 方法：snake_case
    impl UserProfile {
        fn full_name(&self) -> String { self.name.clone() }
    }
}
```

## 5. 下划线变量的特殊含义

### 生活类比 🎯

> `_` 就像一个"垃圾桶"——你把东西扔进去，但不关心它是什么。编译器也明白：下划线开头的变量是"故意的忽略"。

### 概念讲解

单下划线 `_` 是一个特殊的"丢弃"模式（discard pattern）。用于告诉编译器：这个值我不需要，不需要警告"未使用变量"。

编译器视角：`_` 在模式匹配中被编译器特殊处理，不绑定到任何变量，因此不会产生"未使用变量"的警告。

### 代码示例

```rust
fn underscore_usage() {
    let _unused = 42;        // 普通变量但未使用 → 警告
    let _ignored = "hello"; // 下划线开头 → 编译器直接忽略，不产生警告

    // 模式匹配中丢弃不需要的值
    let tuple = (1, 2, 3);
    let (_, second, _) = tuple; // 只取第二个值
    println!("second = {}", second);

    // 在解构中忽略字段
    struct Point { x: i32, y: i32 }
    let pt = Point { x: 10, y: 20 };
    let Point { x: _, y: _ } = pt; // 忽略所有字段

    // 多次使用下划线
    let (_, _, third) = (10, 20, 30); // 三次下划线分别丢弃三个值
    println!("third = {}", third);
}
```

### 常见错误

```rust
// ❌ 错误：下划线不是变量名，不能用于赋值
// let _ = 5;
// _ = 6; // ❌ 这不是赋值，是创建新的下划线模式

// ✅ 正确：需要变量名时用 _name（带名字的下划线）
let _value = 42;
let x = _value + 1; // ✅ 可以使用
```

## 本章小结

| 特性 | Rust 行为 |
|------|-----------|
| 行注释 | `//` 编译器忽略 |
| 块注释 | `/* */` 编译器忽略 |
| 文档注释 | `///` 会编译进 rustdoc |
| 缩进 | 不强制，但建议 4 空格 |
| 关键字 | 不能直接用作标识符，`r#` 前缀可转义 |
| 下划线变量 | `_` 表示丢弃，不触发未使用警告 |
| 命名规范 | snake_case(变量), PascalCase(类型), SCREAMING(常量) |
